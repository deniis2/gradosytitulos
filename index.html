VISTAS
Views

Una vista es una tabla virtual cuyo contenido se define mediante una consulta SELECT. No almacena datos propios — actúa como una ventana hacia una o más tablas base. Permite simplificar consultas, mejorar la seguridad y abstraer la complejidad del esquema.

CREATE VIEW — Sintaxis completa en SQL Server

T-SQL · SQL Server 2019+ · Fuente: docs.microsoft.com
-- ──────────────────────────────────────────────────────────── -- SINTAXIS GENERAL DE CREATE VIEW -- ──────────────────────────────────────────────────────────── CREATE OR ALTER VIEW schema_name.view_name [ ( column_name [ ,...n ] ) ] [ WITH <view_attribute> [ ,...n ] ] AS SELECT -- consulta de definición [ WITH CHECK OPTION ]; -- ──────────────────────────────────────────────────────────── -- EJEMPLO 1: Vista simple — un solo SELECT con JOIN -- ──────────────────────────────────────────────────────────── CREATE OR ALTER VIEW dbo.vw_EmpleadosActivos AS SELECT e.EmpleadoID, e.Nombres + ' ' + e.Apellidos AS NombreCompleto, d.NombreDepto AS Departamento, e.Salario, e.FechaIngreso FROM Empleados e JOIN Departamentos d ON e.DepID = d.DepID WHERE e.Estado = 'Activo'; GO -- Consultar la vista exactamente como una tabla SELECT * FROM dbo.vw_EmpleadosActivos WHERE Departamento = 'Ingeniería' ORDER BY NombreCompleto; GO -- ──────────────────────────────────────────────────────────── -- EJEMPLO 2: Vista con ENCRYPTION y SCHEMABINDING -- WITH ENCRYPTION → protege el código fuente de la vista -- WITH SCHEMABINDING → impide modificar/eliminar tablas base -- ──────────────────────────────────────────────────────────── CREATE OR ALTER VIEW dbo.vw_SalariosConfidenciales WITH ENCRYPTION, SCHEMABINDING AS SELECT e.EmpleadoID, e.Salario, e.NivelSalarial FROM dbo.Empleados e; -- SCHEMABINDING requiere schema.table GO -- ──────────────────────────────────────────────────────────── -- EJEMPLO 3: Vista indexada (materializada) — mejora rendimiento -- Requiere SCHEMABINDING + índice CLUSTERED UNIQUE -- ──────────────────────────────────────────────────────────── CREATE OR ALTER VIEW dbo.vw_TotalVentasPorProducto WITH SCHEMABINDING AS SELECT ProductoID, SUM(Cantidad) AS TotalCantidad, SUM(Importe) AS TotalImporte, COUNT_BIG(*) AS NumTransacciones FROM dbo.DetalleVentas GROUP BY ProductoID; GO -- Crear índice físico sobre la vista (la materializa en disco) CREATE UNIQUE CLUSTERED INDEX IX_vw_TotalVentas_Producto ON dbo.vw_TotalVentasPorProducto (ProductoID); GO -- ──────────────────────────────────────────────────────────── -- GESTIÓN DE VISTAS -- ──────────────────────────────────────────────────────────── ALTER VIEW dbo.vw_EmpleadosActivos AS SELECT ...; -- modificar DROP VIEW dbo.vw_EmpleadosActivos; -- eliminar SELECT * FROM sys.views; -- listar vistas EXEC sp_helptext 'dbo.vw_EmpleadosActivos'; -- ver definición
Tipos de vistas

Los 3 tipos que ofrece SQL Server

🔍
Vista Simple
Basada en una sola tabla. No usa GROUP BY ni funciones de agregado. Permite operaciones DML (INSERT, UPDATE, DELETE) directamente sobre la vista, que se reflejan en la tabla base.
DML permitido
🔗
Vista Compleja
Involucra múltiples tablas con JOINs, GROUP BY, HAVING o funciones de agregado (SUM, COUNT…). Es de solo lectura — el motor no puede inferir cómo actualizar las tablas base sin ambigüedad.
Solo lectura
⚡
Vista Indexada (Materializada)
Los resultados del SELECT se almacenan físicamente en disco con un índice CLUSTERED UNIQUE. Consultas muy rápidas porque evitan recalcular el JOIN/agregado. Requiere WITH SCHEMABINDING.
Datos en disco
Cláusulas WITH opcionales

Modificadores de comportamiento de la vista

CláusulaQué haceCuándo usarRestricciones
WITH ENCRYPTION Cifra el texto de definición en sys.syscomments. No se puede leer con sp_helptext Código propietario o sensible que no debe verse No se puede descifrar fácilmente; guarda el original
WITH SCHEMABINDING Vincula la vista a las tablas base: impide DROP TABLE o ALTER TABLE que elimine columnas usadas Antes de crear una vista indexada (requisito) Referencias deben incluir schema: dbo.Tabla
WITH CHECK OPTION Garantiza que toda inserción/actualización a través de la vista cumpla la cláusula WHERE de la vista Cuando la vista tiene WHERE y se hacen DML Solo vistas simples actualizables
WITH VIEW_METADATA Devuelve metadatos de la vista (no de las tablas base) al driver ODBC/OLE DB Aplicaciones que usan la vista como si fuera tabla Afecta la información de esquema devuelta
Ventajas clave

¿Por qué usar vistas en producción?

1
Seguridad granular — ocultar columnas sensibles
Crea una vista sin las columnas de salario, contraseña o DNI. Otorga SELECT sobre la vista en lugar de la tabla. El usuario solo ve las columnas permitidas — nunca accede a la tabla directamente.
GRANT SELECT ON dbo.vw_EmpleadosPublicos TO UsuarioReporte;
2
Abstracción — encapsular complejidad
Un JOIN de 5 tablas con 3 condiciones WHERE queda encapsulado en un nombre legible. Las aplicaciones consultan vw_ResumenPedido sin conocer la estructura interna del esquema.
SELECT * FROM vw_ResumenPedido WHERE PedidoID = 1001; -- simple para el dev
3
Independencia lógica — cambios sin romper aplicaciones
Si se refactoriza la estructura de tablas (dividir Clientes en ClienteNacional y ClienteInternacional), se actualiza la vista y las aplicaciones siguen funcionando sin cambios en su código SQL.
Modifica la vista → las apps que la consumen no cambian ✓
4
Rendimiento con vistas indexadas
Para agregaciones pesadas (SUM de millones de filas), una vista indexada precalcula y almacena los resultados. SQL Server puede usar ese índice automáticamente incluso en consultas sobre las tablas base (NOEXPAND hint).
SELECT * FROM dbo.vw_TotalVentas WITH (NOEXPAND); -- usa el índice físico
Cláusulas opcionales
WITH ENCRYPTION WITH SCHEMABINDING WITH CHECK OPTION WITH VIEW_METADATA CREATE OR ALTER VIEW sys.views sp_helptext
💡
Las vistas NO almacenan datos — son consultas guardadas
Cada vez que consultas una vista simple, SQL Server ejecuta su SELECT internamente. La excepción son las vistas indexadas (WITH SCHEMABINDING + índice CLUSTERED UNIQUE), que sí almacenan los datos físicamente en disco para máximo rendimiento.

FUNCIONES
User-Defined Functions

Las funciones definidas por el usuario (UDF) son bloques de código T-SQL que aceptan parámetros, contienen lógica y retornan un valor escalar o una tabla. A diferencia de los Stored Procedures, pueden usarse directamente dentro de un SELECT.

Tipo 1 — Función Escalar

Retorna un único valor: INT, VARCHAR, DECIMAL, DATE, etc.

T-SQL · Scalar Function · SQL Server 2019+
-- ──────────────────────────────────────────────────────────── -- FUNCIÓN ESCALAR: calcula precio con IGV (18%) -- Retorna: un solo valor DECIMAL -- ──────────────────────────────────────────────────────────── CREATE OR ALTER FUNCTION dbo.fn_PrecioConIGV ( @PrecioBase DECIMAL(10,2), @PorcentajeIGV DECIMAL(4,2) = 18.00 -- parámetro con DEFAULT ) RETURNS DECIMAL(10,2) AS BEGIN DECLARE @Resultado DECIMAL(10,2); SET @Resultado = @PrecioBase * ( 1 + @PorcentajeIGV / 100 ); RETURN @Resultado; END; GO -- Uso directo en SELECT — sintaxis con schema obligatorio SELECT Producto, PrecioBase, dbo.fn_PrecioConIGV(PrecioBase, 18) AS PrecioFinal FROM Catalogo; GO
Tipo 2 — Función de Tabla en Línea (Inline TVF)

Retorna una tabla — la opción más eficiente para devolver conjuntos de datos

T-SQL · Inline Table-Valued Function (iTVF)
-- ──────────────────────────────────────────────────────────── -- INLINE TVF: retorna ventas filtradas de un cliente -- No tiene BEGIN/END — solo RETURN (SELECT...) -- El optimizador puede "expandirla" = muy eficiente -- ──────────────────────────────────────────────────────────── CREATE OR ALTER FUNCTION dbo.fn_VentasPorCliente ( @ClienteID INT, @FechaDesde DATE ) RETURNS TABLE AS RETURN ( SELECT v.VentaID, v.Fecha, v.Subtotal, v.IGV, v.Total, p.NombreProducto FROM Ventas v JOIN DetalleVentas d ON v.VentaID = d.VentaID JOIN Productos p ON d.ProductoID = p.ProductoID WHERE v.ClienteID = @ClienteID AND v.Fecha >= @FechaDesde ); GO -- Uso con CROSS APPLY — combina cada cliente con sus ventas SELECT c.Nombre, v.VentaID, v.Fecha, v.Total FROM Clientes c CROSS APPLY dbo.fn_VentasPorCliente( c.ClienteID, '2024-01-01' ) v; GO
Tipo 3 — Función Multi-sentencia (MSTVf)

Tabla retornada con lógica imperativa (variables, IF, WHILE)

T-SQL · Multi-Statement TVF (MSTVf)
-- ──────────────────────────────────────────────────────────── -- MULTI-STATEMENT TVF: construye la tabla con lógica compleja -- Usa @tabla como variable de retorno — luego RETURN -- ──────────────────────────────────────────────────────────── CREATE OR ALTER FUNCTION dbo.fn_EmpleadosJerarquia ( @GerenteID INT ) RETURNS @Resultado TABLE ( EmpleadoID INT, Nombre VARCHAR(100), Nivel INT, GerenteDirecto INT ) AS BEGIN -- Insertar el gerente raíz INSERT INTO @Resultado SELECT EmpleadoID, Nombre, 0, NULL FROM Empleados WHERE EmpleadoID = @GerenteID; -- Insertar subordinados directos (nivel 1) INSERT INTO @Resultado SELECT EmpleadoID, Nombre, 1, @GerenteID FROM Empleados WHERE GerenteID = @GerenteID; RETURN; END; GO

Restricciones importantes de las UDF
✗ Lo que NO pueden hacer las UDF
No pueden ejecutar INSERT / UPDATE / DELETE sobre tablas permanentes. No pueden llamar Stored Procedures. No pueden usar funciones no deterministas como GETDATE() o NEWID() en SCHEMABINDING. No pueden acceder a sys.* ni enviar mensajes RAISERROR.
✓ Lo que SÍ pueden hacer las UDF
Usar SELECT sobre cualquier tabla. Declarar variables locales (@var). Usar IF / ELSE / WHILE / TRY-CATCH. Modificar tablas temporales tipo variable (@tabla TABLE). Llamar otras funciones escalares o TVF deterministas.
⚡
Prefiere Inline TVF sobre Escalar cuando trabajes con muchas filas
La función escalar se ejecuta fila por fila (N llamadas para N filas). El Inline TVF es expandido por el optimizador de SQL Server como si fuera un JOIN — el motor decide el mejor plan de ejecución automáticamente, sin iteración por fila.

ESQUEMAS
Schemas

Un esquema es un contenedor lógico de objetos dentro de una base de datos. Agrupa tablas, vistas, procedimientos y funciones bajo un namespace común, facilitando la organización, la gestión de permisos y la separación de módulos de negocio.

Crear, usar, asignar permisos y mover objetos entre esquemas

T-SQL · Schemas · SQL Server 2019+
-- ──────────────────────────────────────────────────────────── -- 1. CREAR ESQUEMAS (se crean antes que sus objetos) -- ──────────────────────────────────────────────────────────── CREATE SCHEMA Ventas AUTHORIZATION dbo; GO CREATE SCHEMA RRHH AUTHORIZATION dbo; GO CREATE SCHEMA Finanzas AUTHORIZATION dbo; GO CREATE SCHEMA Auditoria AUTHORIZATION dbo; GO -- ──────────────────────────────────────────────────────────── -- 2. CREAR TABLA DENTRO DE UN ESQUEMA -- Formato: Schema.Tabla (siempre especificar schema) -- ──────────────────────────────────────────────────────────── CREATE TABLE Ventas.Pedidos ( PedidoID INT IDENTITY(1,1) PRIMARY KEY, ClienteID INT NOT NULL, Fecha DATE NOT NULL DEFAULT GETDATE(), Total DECIMAL(10,2) NOT NULL ); GO -- ──────────────────────────────────────────────────────────── -- 3. GESTIÓN DE PERMISOS A NIVEL DE SCHEMA -- Un GRANT sobre el schema aplica a TODOS sus objetos -- ──────────────────────────────────────────────────────────── -- Analista de ventas: solo lectura en módulo Ventas GRANT SELECT ON SCHEMA::Ventas TO AnalstaVentas; -- RRHH bloqueado para usuarios externos DENY SELECT ON SCHEMA::RRHH TO UsuarioExterno; -- Acceso completo DML para el sistema de ventas GRANT INSERT, UPDATE ON SCHEMA::Ventas TO AppVentas; GO -- ──────────────────────────────────────────────────────────── -- 4. MOVER UN OBJETO DE UN SCHEMA A OTRO -- No copia los datos: solo cambia el namespace -- ──────────────────────────────────────────────────────────── -- Mover tabla Empleados de dbo hacia RRHH ALTER SCHEMA RRHH TRANSFER dbo.Empleados; GO -- Mover un procedimiento almacenado ALTER SCHEMA Ventas TRANSFER dbo.usp_RegistrarPedido; GO -- ──────────────────────────────────────────────────────────── -- 5. CONSULTAR SCHEMAS EXISTENTES Y SUS OBJETOS -- ──────────────────────────────────────────────────────────── SELECT schema_name, schema_owner FROM INFORMATION_SCHEMA.SCHEMATA; SELECT s.name AS Esquema, o.name AS Objeto, o.type_desc FROM sys.objects o JOIN sys.schemas s ON o.schema_id = s.schema_id ORDER BY s.name, o.name;

Schema vs. Base de Datos — ¿cuándo usar cada uno?

🗂️ Schema (Esquema)
División lógica dentro de una sola BD
Comparten el mismo motor de transacciones
JOIN entre schemas sin configuración extra
Un GRANT sobre schema aplica a todos sus objetos
Sin costo adicional de recursos del servidor
Ideal: módulos de negocio (Ventas, RRHH, Finanzas)
VS
🗄️ Base de Datos
Entorno completamente aislado y autónomo

Ventajas en producción
🏢 Organización por módulo de negocio
Ventas.Pedidos, RRHH.Empleados, Finanzas.CuentasPagar. En lugar de 50 tablas todas en dbo, cada área tiene su espacio limpio. Los desarrolladores saben exactamente dónde está cada objeto.
🔐 Seguridad simplificada
GRANT SELECT ON SCHEMA::Ventas TO AnalstaVentas otorga acceso a TODAS las tablas, vistas y funciones del schema Ventas en una sola línea, sin necesidad de hacer GRANT objeto por objeto.
🗂️
Usa schemas para separar módulos: Ventas, RRHH, Compras, Finanzas
Un permiso GRANT en el schema se propaga automáticamente a todos sus objetos presentes y futuros. Esto reduce el trabajo de administración y asegura que ningún objeto nuevo quede sin permisos accidentalmente.

ÍNDICES
Indexes

Un índice es una estructura B-Tree en disco que permite a SQL Server localizar filas sin escanear la tabla completa. Reduce la complejidad de búsqueda de O(n) a O(log n), acelerando dramáticamente las consultas SELECT en tablas con millones de filas.

Sintaxis completa

Tipos de índices y cómo crearlos en SQL Server

T-SQL · Indexes · SQL Server 2019+
-- ──────────────────────────────────────────────────────────── -- ÍNDICE CLUSTERED -- Ordena físicamente los datos de la tabla en disco -- Solo puede haber UNO por tabla (la tabla IS el índice) -- El PRIMARY KEY lo crea automáticamente (salvo override) -- ──────────────────────────────────────────────────────────── CREATE CLUSTERED INDEX IX_Empleados_ID ON Empleados ( EmpleadoID ASC ); GO -- ──────────────────────────────────────────────────────────── -- ÍNDICE NON-CLUSTERED (el más común) -- Estructura separada con puntero a la fila real -- Múltiples por tabla — ideal para columnas en WHERE / JOIN -- INCLUDE agrega columnas "cubiertas" sin ser clave del índice -- ──────────────────────────────────────────────────────────── CREATE NONCLUSTERED INDEX IX_Empleados_Apellido ON Empleados ( ApellidoPaterno ASC, Nombres ASC ) INCLUDE ( Salario, DepID, Estado ); -- columnas cubiertas GO -- ──────────────────────────────────────────────────────────── -- ÍNDICE ÚNICO — también aplica constraint de unicidad -- ──────────────────────────────────────────────────────────── CREATE UNIQUE NONCLUSTERED INDEX IX_Empleados_Email ON Empleados ( Email ) WHERE Email IS NOT NULL; -- FILTERED: solo filas con email GO -- ──────────────────────────────────────────────────────────── -- ÍNDICE FILTRADO (Filtered Index) — subconjunto de filas -- Más pequeño, más rápido, consume menos espacio -- ──────────────────────────────────────────────────────────── CREATE NONCLUSTERED INDEX IX_Pedidos_Pendientes ON Pedidos ( FechaPedido, ClienteID ) WHERE Estado = 'Pendiente'; -- solo pedidos pendientes GO -- ──────────────────────────────────────────────────────────── -- MANTENIMIENTO DE ÍNDICES -- Ver fragmentación (>30% REBUILD, 5-30% REORGANIZE) -- ──────────────────────────────────────────────────────────── SELECT OBJECT_NAME(ips.object_id) AS Tabla, i.name AS Indice, ips.avg_fragmentation_in_percent AS Fragmentacion FROM sys.dm_db_index_physical_stats (DB_ID(), NULL, NULL, NULL, 'LIMITED') ips JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id WHERE ips.avg_fragmentation_in_percent > 5 ORDER BY ips.avg_fragmentation_in_percent DESC; GO -- Reconstruir un índice (fragmentación > 30%) ALTER INDEX IX_Empleados_Apellido ON Empleados REBUILD; -- Reorganizar un índice (fragmentación 5%–30%) ALTER INDEX ALL ON Empleados REORGANIZE; GO
Comparativa fundamental

Clustered vs. Non-Clustered — diferencias clave

🏗️ Clustered Index
Solo 1 por tabla — el PK lo crea automáticamente
Ordena físicamente los datos de la tabla en disco
Las páginas de datos y las hojas del índice son lo mismo
Búsquedas por rango (BETWEEN, >, <) muy eficientes
Elegir columna con inserciones secuenciales (IDENTITY)
VS
📑 Non-Clustered Index
Hasta 999 por tabla — múltiples columnas
Estructura separada: hoja contiene puntero a la fila real
Puede incluir columnas adicionales con INCLUDE (covering)
Ideal para columnas frecuentes en WHERE, JOIN, ORDER BY
Filtered index: subconjunto de filas — más pequeño y rápido
Guía de decisión

¿Cuándo crear un índice?

🔎
WHERE frecuente
Columnas que aparecen constantemente en WHERE o HAVING con alta cardinalidad (muchos valores distintos).
🔗
Columnas JOIN
Las columnas usadas en ON para unir tablas grandes. Evitan que SQL Server escanee toda la tabla al hacer el JOIN.
📊
ORDER BY / GROUP BY
Evita operaciones de ordenamiento costosas en memoria (Sort operator en el plan de ejecución).
⚠️
Cuándo NO usar
Tablas pequeñas, columnas con poca selectividad (Género: M/F), tablas con inserciones masivas continuas.
Mantenimiento — Fragmentación de índices
FragmentaciónCausaAcción recomendadaComando
< 5% Normal — páginas bien ordenadas Ninguna acción necesaria —
5% – 30% Inserciones/eliminaciones frecuentes REORGANIZE (en línea, sin bloqueo) ALTER INDEX … REORGANIZE
> 30% Alta fragmentación — búsquedas lentas REBUILD (reconstruye el árbol completo) ALTER INDEX … REBUILD
📊
Los índices aceleran SELECT pero ralentizan INSERT / UPDATE / DELETE
Cada vez que se modifica una fila, SQL Server debe actualizar también todos los índices de esa tabla. Monitorea la fragmentación con sys.dm_db_index_physical_stats y reconstruye cuando supere el 30%. Nunca crees índices "por si acaso" — analiza el Query Execution Plan primero.

Un Stored Procedure es un bloque de código T-SQL precompilado almacenado en la base de datos. Acepta parámetros de entrada y salida, ejecuta cualquier operación DML, maneja transacciones y errores, y puede devolver conjuntos de resultados o valores de retorno.

Stored Procedure profesional — estructura correcta en producción

T-SQL · Stored Procedure · SQL Server 2019+ · Producción
-- ──────────────────────────────────────────────────────────── -- usp_RegistrarVenta — SP completo con: -- · Parámetros de entrada y OUTPUT -- · SET NOCOUNT ON (elimina mensajes "N rows affected") -- · Transacción explícita BEGIN / COMMIT / ROLLBACK -- · Manejo de errores con TRY / CATCH y THROW -- ──────────────────────────────────────────────────────────── CREATE OR ALTER PROCEDURE dbo.usp_RegistrarVenta ( @ClienteID INT, @ProductoID INT, @Cantidad INT, @Descuento DECIMAL(5,2) = 0.00, -- parámetro con DEFAULT @VentaID INT OUTPUT -- parámetro de retorno ) AS BEGIN SET NOCOUNT ON; DECLARE @PrecioUnitario DECIMAL(10,2), @Subtotal DECIMAL(10,2), @IGV DECIMAL(10,2), @Total DECIMAL(10,2); BEGIN TRY BEGIN TRANSACTION; -- Obtener precio del producto SELECT @PrecioUnitario = PrecioVenta FROM Productos WHERE ProductoID = @ProductoID AND Activo = 1; IF @PrecioUnitario IS NULL THROW 50001, 'Producto no existe o está inactivo.', 1; -- Calcular importes SET @Subtotal = @Cantidad * @PrecioUnitario * (1 - @Descuento / 100); SET @IGV = @Subtotal * 0.18; SET @Total = @Subtotal + @IGV; -- Registrar la venta INSERT INTO Ventas ( ClienteID, ProductoID, Cantidad, Subtotal, IGV, Total, Fecha ) VALUES ( @ClienteID, @ProductoID, @Cantidad, @Subtotal, @IGV, @Total, GETDATE() ); -- Capturar el ID generado automáticamente SET @VentaID = SCOPE_IDENTITY(); -- Descontar stock UPDATE Productos SET Stock = Stock - @Cantidad WHERE ProductoID = @ProductoID; COMMIT TRANSACTION; END TRY BEGIN CATCH IF XACT_STATE() <> 0 ROLLBACK TRANSACTION; THROW; -- re-lanza el error al llamante END CATCH END; GO -- ──────────────────────────────────────────────────────────── -- EJECUTAR EL STORED PROCEDURE -- ──────────────────────────────────────────────────────────── DECLARE @NuevoID INT; EXEC dbo.usp_RegistrarVenta @ClienteID = 10, @ProductoID = 5, @Cantidad = 3, @Descuento = 10.00, @VentaID = @NuevoID OUTPUT; SELECT 'Venta registrada con ID: ' + CAST(@NuevoID AS NVARCHAR) AS Resultado; GO
Flujo de ejecución de un SP

¿Qué ocurre internamente cuando se ejecuta EXEC usp_?

1
Parse & Resolución de nombres
SQL Server analiza la sintaxis T-SQL del SP y verifica que todas las tablas, columnas y objetos referenciados existan. Los errores de sintaxis se detectan aquí antes de ejecutar nada.
Si hay error de sintaxis: "Incorrect syntax near …" — el SP no se crea
2
Compilación y optimización del plan de ejecución
El Query Optimizer genera el plan de ejecución óptimo basado en estadísticas de índices y distribución de datos. Este plan se almacena en el Plan Cache de SQL Server para reutilizarlo en ejecuciones posteriores.
Plan cacheado = la 2ª ejecución es más rápida que la 1ª (no recompila)
3
Ejecución con los parámetros recibidos
SQL Server ejecuta el plan usando los valores reales pasados como parámetros. Las variables locales se inicializan, las sentencias DML se ejecutan dentro de la transacción y el manejo de errores TRY/CATCH está activo.
SCOPE_IDENTITY() captura el último IDENTITY insertado en la sesión actual

Retorno de resultados al llamante
El SP puede devolver: conjuntos de resultados (SELECT), parámetros marcados como OUTPUT, el código de retorno RETURN n (0 = éxito), y mensajes de error vía THROW. El llamante decide qué hacer con cada uno.
RETURN 0 = éxito · RETURN -1 = error de negocio · THROW = excepción

⚙ Convención de nombres
Usa el prefijo usp_ (user stored procedure) en lugar de sp_. El prefijo sp_ hace que SQL Server busque el procedimiento primero en la base de datos master antes de buscarlo en la BD actual, generando overhead innecesario en cada llamada.
✓ dbo.usp_RegistrarVenta ← correcto
✗ dbo.sp_RegistrarVenta ← evitar
⚙️
Los Stored Procedures son la capa de lógica de negocio en la base de datos
Encapsulan operaciones completas con validaciones, transacciones y manejo de errores. Las aplicaciones solo ejecutan EXEC usp_NombreSP — nunca tienen acceso directo a las tablas base. Esto aumenta la seguridad, simplifica el mantenimiento y mejora el rendimiento gracias al plan de ejecución cacheado.
